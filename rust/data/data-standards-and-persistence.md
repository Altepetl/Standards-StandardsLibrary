# Rust — Data Standards and Persistence

Covers conventions for storing, accessing, and evolving persistent data in Rust. The Rust data-access landscape is **deliberately diverse**: there is no "JPA equivalent" and no single ORM monopoly. Several high-quality, philosophically different solutions coexist, and the right choice depends on the project.

> **Catalog, not mandate.** SQLx, Diesel, SeaORM, and rusqlite are documented side by side. Selection happens in StandardBuilder.

## 1. Database access — the recognized options

| Solution | Style | Async | SQL | Compile-time checks |
|---|---|---|---|---|
| [`sqlx`](https://crates.io/crates/sqlx) | Query macros / plain SQL | Yes (Tokio / async-std) | Hand-written SQL | ✅ queries checked at build time against a live DB or `sqlx prepare` offline cache |
| [`diesel`](https://crates.io/crates/diesel) | ORM + DSL | Yes (2.x with `async` feature) | DSL that compiles to SQL | ✅ schema-driven; queries checked against `schema.rs` |
| [`sea-orm`](https://crates.io/crates/sea-orm) | Async ORM (built on `sqlx`) | Yes | DSL | ✅ entity models generated from schema |
| [`rusqlite`](https://crates.io/crates/rusqlite) | Synchronous SQLite binding | No | Hand-written SQL | ❌ queries checked at runtime |
| [`tokio-postgres`](https://crates.io/crates/tokio-postgres) | Async PostgreSQL driver | Yes | Hand-written SQL | ❌ runtime |
| [`mysql_async`](https://crates.io/crates/mysql_async) | Async MySQL driver | Yes | Hand-written SQL | ❌ runtime |

### sqlx — the most popular modern choice

`sqlx` is async, framework-light, and lets you write **real SQL** while getting **compile-time** verification that the query parses and the columns match your Rust types:

```rust
use sqlx::postgres::PgPool;

let pool = PgPool::connect("postgres://localhost/shop").await?;

// ✅ Compile-time-checked parameterized query
let users: Vec<User> = sqlx::query_as!(
    User,
    "SELECT id, email, created_at FROM users WHERE created_at > $1",
    since
)
.fetch_all(&pool)
.await?;
```

Compile-time checking requires either a live database at build time (`DATABASE_URL`) or a checked-in `sqlx-data.json` / `.sqlx/` directory generated by `cargo sqlx prepare`. CI uses the offline cache.

### Diesel — schema-driven ORM

`diesel` generates `schema.rs` from your database schema, and its DSL is checked at compile time against that schema:

```rust
use diesel::prelude::*;
use diesel::pg::PgConnection;

let conn = &mut PgConnection::establish("postgres://localhost/shop")?;

let users: Vec<User> = users::table
    .filter(users::created_at.gt(since))
    .select(User::as_select())
    .load(conn)?;
```

Trade-off: stronger type safety and a query DSL, but you give up writing raw SQL and the migration story is more opinionated.

### SeaORM — async ORM

`sea-orm` is built on top of `sqlx` and provides an async, ActiveRecord-style ORM with entity models generated from the schema. Good fit for projects that want an ORM but find Diesel's sync API limiting.

### rusqlite — embedded SQLite

`rusqlite` is the canonical synchronous binding to SQLite. Used for CLIs, desktop apps (Tauri), and any project that ships a single-file database.

```rust
use rusqlite::{params, Connection};

let conn = Connection::open("app.db")?;
conn.execute(
    "INSERT INTO users (email) VALUES (?1)",
    params![&email],
)?;
```

### The "raw driver" layer

`tokio-postgres` (PostgreSQL), `mysql_async` (MySQL), and the lower-level `sqlx-postgres` are useful when you want a driver without an ORM or query macros.

## 2. The cardinal rule: parameterized queries, never string-concatenated SQL

This is true regardless of which crate you pick.

### ❌ String concatenation — SQL injection

```rust
// ❌ NEVER DO THIS:
let q = format!("SELECT id FROM users WHERE email = '{}'", email);
let rows = sqlx::query(&q).fetch_all(&pool).await?;
```

### ✅ Parameter binding

```rust
// sqlx
sqlx::query("SELECT id FROM users WHERE email = $1")
    .bind(&email)
    .fetch_optional(&pool).await?;

// rusqlite
conn.query_row(
    "SELECT id FROM users WHERE email = ?1",
    params![&email],
    |row| row.get(0),
)?;

// diesel — DSL cannot express injection by construction
users::table.filter(users::email.eq(&email)).first::<User>(conn)?;
```

Every recognized Rust database crate binds parameters separately from the SQL text. There is no ergonomic reason ever to concatenate user data into a SQL string.

## 3. Migrations

| Tool | Style | Notes |
|---|---|---|
| `sqlx migrate` | Plain SQL files in `migrations/`, applied in order, recorded in `_sqlx_migrations`. | First-class with sqlx; supports up-only and revert files. |
| `diesel migration` | Plain SQL files in `migrations/`; regenerates `schema.rs`. | The Diesel CLI manages them. |
| [`refinery`](https://crates.io/crates/refinery) | Embedded migrations (SQL or Rust), versioned. | Driver-agnostic; works with postgres, mysql, sqlite, tokio-postgres. |
| [`barrel`](https://crates.io/crates/barrel) | Programmatic schema changes in Rust. | Useful for backend-agnostic migrations. |

### Zero-downtime migration convention — expand / contract

The recognized pattern for shipping schema changes without downtime is **expand/contract (parallel change)**, applicable in Rust exactly as in other ecosystems:

```
Phase 1 (expand):   add a nullable column / new table; old code still works.
Phase 2 (backfill): populate the new column for existing rows (background job).
Phase 3 (switch):   deploy code that writes AND reads the new column.
Phase 4 (contract): make the column NOT NULL, drop the old path.
```

Each phase is a separate migration + deploy; never batch a breaking change with the code that depends on it.

```sql
-- Migration 1 (expand)
ALTER TABLE orders ADD COLUMN currency_code TEXT;
-- Migration 2 (backfill) — separate step
UPDATE orders SET currency_code = 'USD' WHERE currency_code IS NULL;
-- Migration 3 (contract) — only after the new code is fully deployed
ALTER TABLE orders ALTER COLUMN currency_code SET NOT NULL;
```

## 4. Serialization — serde is the standard

[`serde`](https://crates.io/crates/serde) is the universal serialization framework. Every recognized storage / messaging format has a serde-compatible adapter:

| Format | Crate |
|---|---|
| JSON | [`serde_json`](https://crates.io/crates/serde_json) — the de facto JSON library |
| TOML | [`toml`](https://crates.io/crates/toml) — the Cargo format, also widely used for app config |
| YAML | [`serde_yaml`](https://crates.io/crates/serde_yaml) (archived but still widely used) or [`serde_yml`](https://crates.io/crates/serde_yml) (maintained fork) |
| bincode | [`bincode`](https://crates.io/crates/bincode) — compact Rust-native binary format |
| MessagePack | [`rmp-serde`](https://crates.io/crates/rmp-serde) |
| Protocol Buffers | [`prost`](https://crates.io/crates/prost) — the rust-lang community's protobuf |
| Cap'n Proto | [`capnp`](https://crates.io/crates/capnp) |
| CBOR | [`ciborium`](https://crates.io/crates/ciborium) |
| Avro | [`apache-avro`](https://crates.io/crates/apache-avro) |

```rust
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize)]
struct Order {
    id: u64,
    email: String,
    total_cents: i64,
    created_at: chrono::DateTime<chrono::Utc>,
}

let json = serde_json::to_string(&order)?;
let back: Order = serde_json::from_str(&json)?;

let bytes = bincode::serialize(&order)?;
let back: Order = bincode::deserialize(&bytes)?;
```

## 5. Configuration — the `config` crate

[`config`](https://crates.io/crates/config) is the de facto standard for layered configuration (file + env + CLI), especially when following the 12-factor pattern.

```rust
use config::{Config, File, Environment};

let settings = Config::builder()
    .add_source(File::with_name("config/default"))
    .add_source(File::with_name("config/production").required(false))
    .add_source(Environment::default().separator("__"))
    .build()?;

let db_url: String = settings.get("database.url")?;
```

Convention: a layered cascade — defaults file → environment-specific file → environment variables → CLI flags — with the higher-priority layer overriding the lower.

## 6. Transaction patterns

```rust
// sqlx — explicit transaction
let mut tx = pool.begin().await?;

sqlx::query!("INSERT INTO accounts (id, balance) VALUES ($1, $2)", aid, 100)
    .execute(&mut *tx).await?;

sqlx::query!("UPDATE accounts SET balance = balance - $2 WHERE id = $1", aid, 50)
    .execute(&mut *tx).await?;

tx.commit().await?;     // or `tx.rollback().await?`
```

Conventions:

- Keep transactions **short**; never await network calls (other than the DB) inside a transaction.
- Use **`SERIALIZABLE`** only when you need it; the default (`READ COMMITTED` on Postgres) is correct for most app code.
- The **outbox pattern** (cross-reference Section 26) is the recognized way to publish messages atomically with a DB write — insert into an `outbox` table inside the transaction, then a separate relay publishes to the broker.

## 7. Schema-as-code, seed data, test data

- **Schema-as-code:** migrations under version control (Section 4 — Project Architecture), applied by `sqlx migrate run` / `diesel migration run` / `refinery` at deploy time.
- **Seed data:** `sqlx migrate add -- reversible` scripts or a `seed.sql`; community convention is a separate `seeds/` directory, not a migration.
- **Test data:** the [`fake-rs`](https://crates.io/crates/fake) crate generates realistic random test data (names, emails, addresses, UUIDs).
- **Test databases:** convention is a per-test ephemeral database (Docker / testcontainers via [`testcontainers`](https://crates.io/crates/testcontainers)) — never run integration tests against a shared DB.

## 9. Identifier naming conventions

The community convention for database identifiers diverges from Rust code conventions:

| Element | Convention |
|---|---|
| Tables | plural `snake_case` (`users`, `order_items`) |
| Columns | `snake_case` (`created_at`, `user_id`) |
| Indexes | `idx_<table>_<col1>_<col2>` or `idx_<table>_<purpose>` |
| Foreign keys | `fk_<table>_<ref_table>` |
| Primary keys | `id` (singular) |

This mirrors the SQL ecosystem at large and is independent of Rust's source naming.

## 10. ID, timestamp, monetary conventions

| Concern | Convention |
|---|---|
| IDs | `uuid` crate (`Uuid::new_v4()`); or for sortable IDs, `ulid` / `sqids`. Auto-increment integers acceptable for non-distributed systems. |
| Timestamps | store as `timestamptz` (Postgres) / UTC, never naive; see Section 20. |
| Monetary values | `DECIMAL` / `NUMERIC` column ↔ `rust_decimal::Decimal`; never `FLOAT`. |
| Booleans | native `BOOLEAN` column ↔ Rust `bool`. |
